# Story 49.2: API de variação — `GET /products/:id/variants` dedicado

**Epic:** [Epic 49 — Homologação Tray (checklist oficial · projeto Odoo #1723)](../epics/epic-49-homologacao-tray.md)
**Story ID:** 49.2
**Priority:** 🟠 High
**Status:** Draft
**Type:** ✨ Feature (nova chamada dedicada ao 3º) + 🧪 evidência de homologação
**Security gate:** ✅ Sim — nova chamada a API de terceiro (Tray) carregando `access_token` na querystring; a evidência exige print de console com a URL completa (risco de vazar token → **mascarar**); e o consumo dedicado por-produto multiplica requisições contra o teto de **180 req/min** (coordenar com 49.4). Dispara Phase 4a.
**Estimativa:** M — 1 mapeador puro novo + 1 helper de fetch em `_shared/tray-products.ts` (molde do que já existe), fiação opcional no `crm-tray-sync-products` (flag + caminho de evidência), testes deno do parser. Sem migration, sem tabela/coluna/RPC nova (reusa `tray_mirror.products.variations`, hoje `unknown`).

## 🎯 Contexto

- **O que a Tray pede (tarefa 06 do board Odoo #1723).** Para cada API do checklist, a Tray exige **print do código da aplicação + print do console** mostrando a **URL consumida (completa)**, o **Content-Type** (só se POST/PUT) e o **JSON/payload**. A tarefa 06 é a **API de variação**: `GET {api_host}/products/:id/variants`. O doc de apoio confirma que hoje isso é lacuna (`docs/homologacao-tray-pacote.md:55-57`).

- **Como funciona hoje (embutido, sem chamada dedicada).** A integração **nunca** chama `.../variants`. As variações são lidas **de dentro** da resposta de `/products` no mapeador do espelho: `supabase/functions/_shared/tray-products.ts:96` faz `variations: p.Variant ?? p.variations ?? []` e joga o array bruto em `TrayProductMirror.variations` (tipo `unknown`, declarado em `tray-products.ts:53`). Esse produto vem do sync paginado em `supabase/functions/crm-tray-sync-products/index.ts:47` (`trayGet(sb, tok, "/products", { limit: 50, page })`, `PAGE=50` em `:19`), que desembrulha `data.Products` (`:52`), tira o wrapper `w.Product ?? w` (`:55`) e chama `upsertTrayProduct(sb, p, openaiKey)` (`:57`). Ou seja: **nenhuma requisição a `/products/:id/variants` acontece** — não há o que capturar no console para a tarefa 06.

- **O cliente Tray que vamos reusar.** Toda chamada passa por `trayGet(sb, tok, path, params)` em `supabase/functions/_shared/tray.ts:125-135`: monta `${api_host}${path}?access_token=…&params`, faz `fetch` com timeout de 15s (`TRAY_TIMEOUT_MS`, `tray.ts:9`), e em **401** faz `refreshTrayToken` + **1 retry** (`tray.ts:129-131`). O token é singleton via `getValidToken(sb)` (`tray.ts:61`). **Não existe limitador de req/min** — por isso a coordenação com a 49.4.

- **Solução proposta.** Adicionar em `_shared/tray-products.ts` (1) um **mapeador puro testável** `mapTrayVariants(payload)` → `TrayVariant[]` (com `variant_id`, `ean`, `price`, `promo_price`, `stock`, `available`, `images` [de `VariantImage`], `sku` [de `Sku`], `raw`), reusando os helpers `num`/`intOrNull` (`tray-products.ts:28,34`) e a mesma lógica de URL de `extractImages` (`:20`); e (2) um **helper de fetch dedicado** `fetchProductVariants(sb, tok, productId)` que chama `trayGet(sb, tok, \`/products/\${productId}/variants\`)` e devolve o array mapeado. Fiar esse helper no `crm-tray-sync-products` por um **caminho de evidência** (`{ variants_for_product_id }`) que loga a URL mascarada + o payload e grava o resultado dedicado em `mirror.variations`, e por uma **flag opt-in** (`with_variants`) para o sync completo (gated pelo teto de 180/min da 49.4). O caminho padrão do sync **não muda** (variações seguem vindo embutidas até `with_variants=true`).

- **Alternativas rejeitadas.** (a) *Continuar só com o embutido e "negociar" com a Tray* — rejeitada: a tarefa 06 exige evidência de **consumir a API de variação**; sem a chamada dedicada não há print de console. (b) *Chamar `.../variants` para TODO produto em todo sync, sem flag* — rejeitada: multiplica as requisições (1 extra por produto com variação) e estoura o teto de 180/min antes mesmo da 49.4 existir. (c) *Criar edge nova só para a evidência* — rejeitada: duplica auth/token/CORS; reusar o `crm-tray-sync-products` (que já tem `getValidToken` + `trayGet` + guard de auth) é mais barato e store-agnostic.

## 🔌 Contrato / endpoints

**Endpoint dedicado (leitura — a Heziom LÊ o catálogo da Tray):**

```
GET {api_host}/products/{product_id}/variants?access_token={TOKEN}
```
- **`{api_host}`** vem SEMPRE de `crm.tray_tokens.api_host` (store-agnostic; teste = `https://lojatesteintegracaotray.commercesuite.com.br/web_api`). Nada hardcoded.
- **Método:** GET (sem corpo). **Content-Type:** n/a (é GET — o campo "Content-Type/payload" da homologação só se aplica a POST/PUT).
- **Paginação (opcional):** a Tray aceita `&limit=&page=` também aqui; para variação o volume por produto é pequeno, então a 1ª página basta (documentar; não paginar salvo se `Variants.length == limit`).
- **Variação única (referência, fora do fluxo de sync):** `GET {api_host}/products/{product_id}/variants/{variant_id}?access_token=…`.

**Payload de resposta esperado (forma Tray — validar contra a loja 1501119 antes de fechar):**

```jsonc
{
  "Variants": [
    {
      "Variant": {
        "id": "998877",
        "product_id": "42",
        "ean": "7899137012345",
        "price": "89.90",
        "promotional_price": "69.90",
        "stock": "5",
        "available": "1",
        "VariantImage": [ { "https": "https://loja/…/v1.jpg", "http": "http://loja/…/v1.jpg" } ],
        "Sku": [ { "type": "Cor", "value": "Azul" }, { "type": "Tamanho", "value": "M" } ]
      }
    }
  ]
}
```

**Caminho de evidência na edge `crm-tray-sync-products` (POST autenticado — admin OU service_role OU cron, guard já existente):**

```jsonc
// Request — caminho de evidência da tarefa 06
{ "variants_for_product_id": "42" }
```
```jsonc
// 200 OK — array mapeado + URL mascarada para a captura de console
{ "ok": true, "product_id": "42", "count": 1, "url_masked":
  "GET https://…/web_api/products/42/variants?access_token=***", "variants": [ { "variant_id": "998877", "ean": "7899137012345", "price": 89.9, "promo_price": 69.9, "stock": 5, "available": true, "images": ["https://loja/…/v1.jpg"], "sku": [ {"type":"Cor","value":"Azul"}, {"type":"Tamanho","value":"M"} ] } ] }
```
```jsonc
// Request — sync completo com variações dedicadas (opt-in; respeita 49.4)
{ "page": 1, "max_pages": 20, "with_variants": true }
```

## 🎯 Critérios de Aceite

- [ ] **CA1** — Novo mapeador puro `mapTrayVariants(payload): TrayVariant[]` em `_shared/tray-products.ts`: desembrulha `payload.Variants ?? payload.variants ?? []`, tira o wrapper `w.Variant ?? w` (mesmo padrão de `w.Product ?? w` do sync) e mapeia cada item via `mapTrayVariant(v)`. Sem estado, sem I/O, sem `any` no contrato público (`raw: unknown`), imports absolutos.
- [ ] **CA2** — `TrayVariant` e `mapTrayVariant` cobrem: `variant_id: String(v.id ?? v.variant_id ?? "")`, `product_id`, `ean: v.ean ?? v.gtin ?? null`, `price: num(v.price)`, `promo_price` (promo `> 0` senão `null`, mesma regra de `trayProductToMirror`), `stock: intOrNull(v.stock)`, `available` (`"1"`/`true`, senão por estoque, igual ao produto), `images: extractVariantImages(v)` (de `VariantImage`/`images`, reusando a lógica de URL de `extractImages`), `sku: mapSkuAttrs(v)` (de `v.Sku ?? v.sku ?? []` → `{type,value}[]`), `raw: v`.
- [ ] **CA3** — Helper de fetch dedicado `fetchProductVariants(sb, tok, productId): Promise<TrayVariant[]>` chama `trayGet(sb, tok, \`/products/\${productId}/variants\`)` (reusa 401→refresh+retry e timeout de 15s de `tray.ts`) e devolve `mapTrayVariants(...)`. `productId` é interpolado no path só como dígitos (`String(Number(productId))` ou guard equivalente) para não injetar querystring/segmento no path.
- [ ] **CA4** — `crm-tray-sync-products` ganha o caminho de evidência `{ variants_for_product_id }`: quando presente, chama `fetchProductVariants`, **loga no console** a URL completa com `access_token` **mascarado** (`?access_token=***`) e o `count`+1º variant, grava o array dedicado em `tray_mirror.products.variations` daquele `tray_id`, e retorna o JSON do contrato acima. Nenhum outro produto é sincronizado nesse caminho.
- [ ] **CA5** — Flag `with_variants` (default **false**) no sync completo: quando `true`, para cada produto **que tem variação** (payload trouxe `Variant` não-vazio OU `has_variation` truthy), chama `fetchProductVariants` e usa o resultado dedicado como `variations` do espelho (sobrepondo o embutido de `tray-products.ts:96`). Quando `false`, o comportamento é **idêntico ao de hoje** (variações embutidas; zero requisição extra).
- [ ] **CA6** — **Segurança (token):** em nenhum caminho o `access_token` é logado em claro. O `url_masked` da resposta e todo `console.log`/`console.error` mascaram o token (`access_token=***`); a URL real só existe dentro do `trayGet`. `reqId = crypto.randomUUID().slice(0,8)` no topo do handler; erros via RFC7807 (`_shared/errors.ts`).
- [ ] **CA7** — **Store-agnostic:** `api_host` sempre de `getValidToken(sb)`/`crm.tray_tokens`; nada aponta para `1501119`/`1345958` no código. A mesma chamada funciona em teste e produção só trocando o token do singleton.
- [ ] **CA8** — Testes deno do parser (ver Testes): `deno test supabase/functions/_shared/tray-variants_test.ts` verde, incluindo o payload real coletado da loja 1501119.
- [ ] **CA9** — Gates verdes: `deno check supabase/functions/{crm-tray-sync-products,_shared}/*.ts` + `deno test supabase/functions/_shared/tray-variants_test.ts`. Sem migration → sem bateria SQL obrigatória.
- [ ] **CA10** — **Evidência anexada à tarefa 06** no board Odoo #1723: print do código (`_shared/tray-products.ts` do mapeador/fetch) + print do console (URL completa com token mascarado + payload/JSON mapeado) rodado contra a loja 1501119. Só então a story vira "Done".

## ⚠️ Riscos

1. **Teto de 180 req/min.** `with_variants=true` no catálogo inteiro = 1 requisição extra por produto com variação → pode estourar o limite. **Mitigação:** flag default false + só ligar após a 49.4; a evidência (CA4) é 1 única chamada por produto escolhido.
2. **Vazamento de token no print.** A URL real leva `access_token` na querystring. **Mitigação (CA6):** `url_masked` e todos os logs mascaram (`access_token=***`); a URL crua só existe dentro do `trayGet`.
3. **Forma do payload divergir do documentado.** Campos de variação variam por versão da API Tray (mesma nota de `tray-products.ts:6-8`). **Mitigação:** mapeador defensivo com fallbacks + guardar `raw` + fixture real da 1501119 no teste (Task 4) antes de fechar.
4. **Injeção no path via `productId`.** `productId` entra no path (`/products/${id}/variants`). **Mitigação (CA3):** normalizar para dígitos antes de interpolar; nunca passar string arbitrária ao path.
5. **Divergência embutido × dedicado.** O array dedicado pode diferir do embutido de `/products`. É esperado (o dedicado é a fonte de verdade da tarefa 06); só sobrepõe `variations` quando `with_variants=true`, sem afetar preço/estoque do produto na projeção `crm.products`.
6. **401 em cadeia no meio do sync.** Coberto: `trayGet` já faz refresh + 1 retry; o token singleton é atualizado para as chamadas seguintes.
