## 🎯 Contexto

A Tray abriu a homologação oficial (projeto Odoo **#1723**, ticket **#1842904**, Marcos M. — Suporte Técnico). Para **cada API do checklist** a Tray exige, por tarefa: **print do código da aplicação + print do console** com a **URL consumida completa**, o **Content-Type** (só quando POST/PUT) e o **JSON/payload**.

A **Tarefa 04 — API de categoria** é a alvo desta story. Hoje a Heziom **não faz** uma chamada dedicada a `/categories`: a categoria do produto chega **embutida** na resposta de `/products` e é lida como texto no mapeador —
`supabase/functions/_shared/tray-products.ts:91` → `category: p.category_name ?? p.Category?.name ?? p.category ?? null`. O sync de catálogo consome só `GET /products?limit=50&page=N` em `supabase/functions/crm-tray-sync-products/index.ts:47`. Ou seja: **não existe evidência de console de um `GET /categories`** para anexar à Tarefa 04, e não guardamos a árvore de categorias (id, pai, hierarquia) — só o nome solto.

Esta story implementa a **chamada dedicada** `GET {api_host}/categories` (lista paginada) e `GET {api_host}/categories/:id` (detalhe) reusando o cliente e o token existentes (`getValidToken`/`trayGet` em `_shared/tray.ts`), **persiste** a árvore em `tray_mirror.categories`, **relaciona** categoria↔produto por id, e gera a **evidência** de homologação (URL/Content-Type/payload). Store-agnostic: `api_host` sempre vem do `/auth` (nunca hardcoded).

> A integração, para categoria, é de **leitura (GET)** — logo o Content-Type/payload da evidência "só se aplica em caso de POST/PUT". A evidência da Tarefa 04 é o **GET com a URL completa + o JSON de resposta**. `POST/PUT/DELETE /categories` ficam fora de escopo (a Heziom não cria categoria na Tray).

## 🎯 Critérios de Aceite

- [ ] **CA1** — Existe uma chamada dedicada `GET {api_host}/categories?access_token=***&limit=50&page=N` disparada pelo código da Heziom (não mais só embutida no `/products`), reusando `getValidToken` + o cliente Tray. Rodando na loja teste **1501119** retorna 200 com a lista de categorias.
- [ ] **CA2** — Existe `GET {api_host}/categories/:id?access_token=***` (detalhe de uma categoria) e retorna 200 para um `id` válido da loja teste.
- [ ] **CA3** — Um **parser puro** (`trayCategoryToMirror`) mapeia a resposta da Tray para um registro normalizado `{ tray_id, name, parent_tray_id, has_products, slug, description, active, raw }`, com fallbacks defensivos para variação de nome de campo entre versões da API (mesmo padrão do `tray-products.ts`) e **sempre** preservando o `raw`.
- [ ] **CA4** — As categorias são **persistidas** em `tray_mirror.categories` (espelho fiel) via upsert idempotente por `tray_id` (re-sync não duplica), com RLS (escrita service_role, leitura `authenticated`) igual ao padrão de `tray_mirror.products`.
- [ ] **CA5** — Há **relação** categoria↔produto: `tray_mirror.products` ganha `category_tray_id` preenchido no sync (a partir do id de categoria vindo no `/products`), e a hierarquia de categoria é navegável por `parent_tray_id` (self-reference).
- [ ] **CA6** — A paginação encerra corretamente: para quando a página volta vazia ou < `limit`, e trata `404` como fim (mesmo contrato de `crm-tray-sync-products`). Timeout de 15s e retry-em-401 herdados de `trayGet` (não reimplementar).
- [ ] **CA7** — **Nenhum** `access_token`/`consumer_secret` aparece em log ou na evidência: a URL logada é **mascarada** (`access_token=***`). Verificado por teste e por grep no diff.
- [ ] **CA8** — **Store-agnostic**: nada de `1501119`, host ou id de loja hardcoded no código; `api_host` sempre do `tray_tokens` (via `getValidToken`).
- [ ] **CA9** — Evidência da Tarefa 04 gerada e anexável ao board Odoo #1723: print do código do helper/sync + print do console com a **URL completa** de `GET /categories` (token mascarado) e o **JSON** de resposta (amostra). Reaproveitável pela Story 49.6.
- [ ] **CA10** — `deno check` verde nas functions tocadas e testes deno do parser passando (ver ## Testes). Security Gate **PASS/CONCERNS** antes do merge.

## 📦 Escopo

**IN:**
- Helper de leitura de categorias reusando `trayGet` (sem novo cliente HTTP, sem novo fluxo de token).
- Parser puro `trayCategoryToMirror` + `upsertTrayCategory` em novo módulo `_shared/tray-categories.ts` (espelha `tray-products.ts`).
- Nova edge function `crm-tray-sync-categories` (paginada, incremental, idempotente) — auth admin OU service_role OU cron, igual `crm-tray-sync-products`.
- Migration idempotente: `tray_mirror.categories` (+ RLS) e coluna `tray_mirror.products.category_tray_id`.
- Preenchimento de `category_tray_id` no upsert de produto (ajuste pontual em `tray-products.ts`, sem quebrar o campo `category` textual existente).
- Testes deno do parser (`_shared/tray-categories_test.ts`).
- Geração da evidência (URL/console) para a Tarefa 04.

**OUT:**
- `POST/PUT/DELETE /categories` (a Heziom não escreve categoria na Tray) — fora do checklist de leitura.
- Variações (49.2), imagens (49.3), limitador 180 req/min (49.4 — esta story **consome** o limitador quando ele existir, mas não o implementa), nota fiscal (49.5).
- Projeção de categoria em `crm.*` / UI de catálogo / embedding de categoria. O espelho `tray_mirror.categories` é suficiente para a homologação e para relacionar produto↔categoria; enriquecer é 38.9.
- Backfill retroativo de `category_tray_id` em produtos já sincronizados: se necessário, é um re-sync de produtos (idempotente) — não uma migração de dados nesta story.

## 🔌 Contrato / endpoints (Tray — `developers.tray.com.br`)

Base: `{api_host}` (por loja, vinda do `/auth`; teste = `https://lojatesteintegracaotray.commercesuite.com.br/web_api`).

| Uso | Método | URL (completa, token mascarado) | Content-Type | Payload/params |
|-----|--------|--------------------------------|--------------|----------------|
| Listar categorias | `GET` | `{api_host}/categories?access_token=***&limit=50&page=1` | — (GET) | query: `limit`, `page` |
| Detalhe de categoria | `GET` | `{api_host}/categories/:id?access_token=***` | — (GET) | path: `:id` |

**Forma da resposta (a confirmar contra a resposta real na loja teste — mesma cautela da Story 25.1 com `/products`):** por analogia com `/products` (`data.Products[].Product`), espera-se um envelope
`{ "Categories": [ { "Category": { ... } } ], "paging": { "total": N, "page": 1, "offset": 50 } }`. O parser é **defensivo**: aceita `data.Categories` ou `data.categories`, e cada item como `w.Category ?? w`. Campos esperados do objeto categoria (com fallback): `id`/`category_id`, `name`, `category_parent_id`/`parent_id`, `has_products`, `slug`, `active`, `description`.

**Reuso do cliente (não reimplementar):**
- `trayGet(sb, tok, path, params)` — `supabase/functions/_shared/tray.ts:125`: monta `${api_host}${path}?access_token=...&<params>`, timeout 15s (`TRAY_TIMEOUT_MS`), **refresh + 1 retry em 401**.
- `getValidToken(sb)` — `_shared/tray.ts:60`: token singleton (`crm.tray_tokens`, id=1); refresca se faltam <5min.
- `refreshTrayToken` — `_shared/tray.ts:36`: refresh de uso único (consumer_key/secret dos secrets).

> Chamadas: `await trayGet(sb, tok, "/categories", { limit: 50, page })` e `await trayGet(sb, tok, `/categories/${id}`)`.

## 🗄️ Modelo de dados (migration idempotente — timestamp)

Arquivo novo: `supabase/migrations/<timestamp>_tray_mirror_categories.sql` (usar `supabase migration new tray_mirror_categories`; **nunca** prefixo numérico de 4 dígitos).

```sql
-- Espelho fiel das categorias Tray (Bronze). Padrão de tray_mirror.products (17.9).
CREATE TABLE IF NOT EXISTS tray_mirror.categories (
  id             UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  tray_id        TEXT NOT NULL UNIQUE,
  name           TEXT,
  parent_tray_id TEXT,                    -- self-reference (hierarquia); NULL = raiz
  has_products   BOOLEAN,
  slug           TEXT,
  description    TEXT,
  active         BOOLEAN,
  raw            JSONB DEFAULT '{}',
  synced_at      TIMESTAMPTZ NOT NULL DEFAULT now(),
  created_at     TIMESTAMPTZ NOT NULL DEFAULT now(),
  updated_at     TIMESTAMPTZ NOT NULL DEFAULT now()
);
CREATE INDEX IF NOT EXISTS idx_tray_categories_parent ON tray_mirror.categories(parent_tray_id);

-- RLS: escrita só service_role/webhook; leitura authenticated (idêntico a 0018/17.9).
ALTER TABLE tray_mirror.categories ENABLE ROW LEVEL SECURITY;
DROP POLICY IF EXISTS service_role_bypass ON tray_mirror.categories;
CREATE POLICY service_role_bypass ON tray_mirror.categories FOR ALL USING (true) WITH CHECK (true);
DROP POLICY IF EXISTS read_auth ON tray_mirror.categories;
CREATE POLICY read_auth ON tray_mirror.categories FOR SELECT TO authenticated USING (true);

-- Relação categoria↔produto: id da categoria no espelho de produtos.
ALTER TABLE tray_mirror.products
  ADD COLUMN IF NOT EXISTS category_tray_id TEXT;
CREATE INDEX IF NOT EXISTS idx_tray_products_category ON tray_mirror.products(category_tray_id);

COMMENT ON TABLE tray_mirror.categories IS
  'Epic 49 (49.1): espelho fiel das categorias Tray (GET /categories). Homologação Odoo #1723 Tarefa 04.';
```

> Validar localmente em Postgres isolado com role `service_role` (ver `reference_heziomos_migration_local_validacao`) — o CI não aplica o SQL no PR. O `db push` roda automático no merge para `main`, por isso tudo `IF NOT EXISTS`.

```ts
import { assertEquals } from "https://deno.land/std@0.224.0/assert/mod.ts";
import { trayCategoryToMirror } from "./tray-categories.ts";

Deno.test("mapeia categoria com envelope Category e campos canônicos", () => {
  const m = trayCategoryToMirror({
    id: "12", name: "Bíblias", category_parent_id: "0",
    has_products: "1", slug: "biblias", active: "1", description: "<p>Linha</p>",
  });
  assertEquals(m.tray_id, "12");
  assertEquals(m.name, "Bíblias");
  assertEquals(m.parent_tray_id, null);      // "0" (raiz) → null
  assertEquals(m.has_products, true);
  assertEquals(m.active, true);
});

Deno.test("fallbacks de nome de campo entre versões da API", () => {
  const m = trayCategoryToMirror({ category_id: "77", parent_id: "12", name: "Infantis" });
  assertEquals(m.tray_id, "77");
  assertEquals(m.parent_tray_id, "12");
});

Deno.test("sempre preserva raw e nunca perde o payload", () => {
  const raw = { id: "5", name: "X", campo_desconhecido: 42 };
  assertEquals(trayCategoryToMirror(raw).raw, raw);
});

Deno.test("id ausente/lixo → tray_id vazio (sync ignora)", () => {
  assertEquals(trayCategoryToMirror({}).tray_id, "");
  assertEquals(trayCategoryToMirror({ name: "sem id" }).tray_id, "");
});
```

## ⚠️ Riscos

- **Forma da resposta divergente** — o envelope real de `/categories` pode não ser `{Categories:[{Category}]}` como em `/products`. Mitigação: parser defensivo + validar contra a resposta real na loja teste antes de dar Done (mesmo aprendizado da 25.1); o `raw` garante re-mapeamento barato.
- **`category_id` pode não vir no `/products`** — se o payload de produto trouxer só o nome da categoria (não o id), `category_tray_id` fica NULL. Mitigação: fallbacks múltiplos; a relação por id é *best-effort* — o `category` textual continua sendo a fonte hoje usada; casar por nome fica para 38.9 se necessário.
- **Limite de 180 req/min (Tarefa 08 / Story 49.4)** — categorias costumam ser poucas páginas, mas somadas ao sync de produtos podem se aproximar do teto. Mitigação: quando o limitador da 49.4 existir, esta função passa por ele automaticamente (mesmo `trayGet`); enquanto não existe, manter `max_pages` conservador.
- **Vazamento de token na evidência** — risco recorrente (a Graph API já ecoou token em erro — ver `feedback_heziomos_meta_error_leaks_token`). Mitigação: mascarar `access_token=***` no log/print (CA7) e nunca logar `raw` de `/auth`.
- **Refresh de uso único** — `trayGet` já trata 401 → refresh + salva o novo par (`_shared/tray.ts:36-58`); não abrir um segundo caminho de token nesta story (dono único do refresh é premissa do E38 / Story 38.2).
