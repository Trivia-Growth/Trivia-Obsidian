## 🎯 Contexto

A **Tarefa 10** do board de homologação (Odoo #1723, ticket #1842904 — Marcos M., Suporte Técnico) pede que a integração declare as **"demais APIs consumidas"** — cada uma em **uma ordem separada** — e adverte: *"ex.: API de script, API de MultiCD; script externo exige detalhamento técnico completo"*. Ou seja, tudo o que a integração toca **além** das tarefas 01–09 (token, categoria, produtos, variação, imagem, limite, NF, webhook, pedidos) precisa ser inventariado, e **qualquer script externo** que consuma a API da Tray precisa de **detalhamento técnico completo** (arquitetura, endpoints, payloads, dados trafegados).

**O que a integração consome além das tarefas 01–09 (verificado no código):** existe um **script externo** — o repositório **`heziom-api`** (Netlify Functions, domínio `api.editoraheziom.com.br`) — que **não vive no monorepo HeziomOS** e consome a Tray por conta própria. É a **ponte CAPI**: recebe o webhook de pedido da Tray, **relê o pedido na Tray** e dispara um evento de conversão **server-side** para o Meta (Conversions API). Como é um consumidor **separado** do mesmo app+loja Tray, ele é exatamente o "script externo" que a Tray quer detalhado.

**Âncoras reais do script externo (`heziom-api`):**
- `netlify/functions/webhooks-tray.js:88` — handler do webhook **primário** (`POST /webhooks/tray`). `parsePayload` (`:136-147`) aceita **`application/x-www-form-urlencoded` e `application/json`**; em `scope_name === "order" && act === "update"` (`:115-117`) chama `processOrderForCAPI`.
- `netlify/functions/webhooks-tray.js:224` — **relê o pedido na Tray**: `GET {TRAY_API_HOST}/web_api/orders/{orderId}/complete?access_token=***` (com self-heal de token + retry em `:229-237`).
- `netlify/functions/webhooks-tray.js:38-86` — `refreshTrayToken()`: `GET {TRAY_API_HOST}/web_api/auth?consumer_key=***&consumer_secret=***&refresh_token=***` (`:51-57`) e PATCH em `tray_tokens`.
- `netlify/functions/webhooks-tray.js:312-380` — `sendCAPIEvent`: **PII hasheada em SHA256** (`:334-341`) e `POST https://graph.facebook.com/v19.0/{PIXEL_ID}/events` (`:376`). Dedup por pedido em `tray_webhook_log` (`:210-220`); filtro por status aprovado (`:256-260`).
- `netlify/functions/tray-capi.js:38` — **fallback/legado** (`POST /api/tray-capi`); valida `X-Tray-Webhook-Secret` (`:48-55`) e mapeia evento→CAPI (`:19-30`).
- `netlify/functions/tray-token-refresh.js:102-104` + `netlify.toml` (`schedule = "0 * * * *"`) — cron **de hora em hora** que renova o token: `GET .../web_api/auth?...refresh_token=***` (`:42-48`).
- Env (Netlify, server-side): `META_PIXEL_ID` (`297709555050094`), `META_CAPI_TOKEN`, `TRAY_WEBHOOK_SECRET`, `TRAY_API_HOST` (⚠️ **hoje aponta para a loja de teste**), `TRAY_CONSUMER_KEY`/`_SECRET`, `SUPABASE_URL`/`SUPABASE_SERVICE_KEY`.

**Os dois "exemplos" que a Tray cita — e como respondê-los:**
- **"API de script" (Tray Scripts / injeção de pixel-tag na vitrine):** **NÃO é consumida hoje.** A Heziom faz o tracking de conversão **server-side** (CAPI, acima), não injetando pixel de browser via a API de Scripts da Tray. Isso é a **38.11** (scripts/pixels via API — Fase 3, ⚪ Low, não implementada). Resposta de homologação: declarar N/A + o método server-side que usamos no lugar.
- **"API de MultiCD" (estoque por centro de distribuição):** **provavelmente N/A** para a Heziom (operação single-CD). É a **38.10** (Multi-CD documentado como opcional). Resposta: **confirmar com a conta** se o Multi-CD está ligado (liga dispara `store_config/update`); se não, declarar N/A.

Esta story é **majoritariamente documentação e evidência** — reúne o detalhamento técnico do script externo e classifica cada "demais API" (consumida / N/A / mapeada a outra story) para virar **ordens separadas** no board. **Não** implementa Tray Scripts, **não** liga Multi-CD e **não** incorpora a `heziom-api` (isso é 38.11/38.10/38.12).

## 🔌 Contrato / endpoints (do script externo `heziom-api`)

Base do script externo: `https://api.editoraheziom.com.br` (Netlify; redirects em `netlify.toml`). `{TRAY_API_HOST}` vem da env `TRAY_API_HOST` (⚠️ **hoje = loja de teste**; store-agnostic é responsabilidade do E38/cutover, não desta story).

### Webhook inbound (Tray → script externo)
```
POST https://api.editoraheziom.com.br/webhooks/tray   (→ webhooks-tray.js:88)
Content-Type: application/x-www-form-urlencoded  OU  application/json   (parsePayload :136-147)
payload (Tray): scope_name=order & scope_id={id} & act=update & seller_id=*** & app_code=***
→ 200 { received: true, scope_name, scope_id, act }
```

### Chamadas do script externo à Tray (evidência da Tarefa 10)
```
GET {TRAY_API_HOST}/web_api/orders/{orderId}/complete?access_token=***      (webhooks-tray.js:224)
Content-Type: n/a (GET)
→ { Order: { id, status, total, Customer:{...}, shipping_address:{...}, ... } }   (relê o pedido p/ montar o evento)

GET {TRAY_API_HOST}/web_api/auth?consumer_key=***&consumer_secret=***&refresh_token=***   (:51-57 / tray-token-refresh.js:42-48)
Content-Type: n/a (GET)
→ { access_token, refresh_token, date_expiration_access_token }   (refresh de USO ÚNICO — grava o novo par)
```

### Destino Meta (não é Tray — é o efeito do script externo)
```
POST https://graph.facebook.com/v19.0/{PIXEL_ID}/events            (webhooks-tray.js:376)
Content-Type: application/json
payload: { data:[{ event_name:"Purchase", event_id:"tray_purchase_{orderId}", user_data:{ em/ph/fn/ln/zp/ct/st (SHA256) }, custom_data:{ currency:"BRL", value, order_id } }], access_token:*** }
```

### Fallback/legado (documentar como existente, não é o caminho primário)
```
POST https://api.editoraheziom.com.br/api/tray-capi               (tray-capi.js:38; valida X-Tray-Webhook-Secret :48-55)
```

> **Content-Type/payload só se aplicam a POST/PUT** (regra do formulário da Tray). As chamadas do script externo **à Tray** são **GET** (`n/a`); o POST com corpo JSON é para o **Meta**, não para a Tray — deixar isso explícito no board para não confundir a evidência.

## ⚠️ Riscos

- **Ambiguidade de "API de script" (risco central):** a Tray pode querer dizer a **Tray Scripts API** (injetar pixel na vitrine) **ou** apenas o **script externo** que consome a API (nossa `heziom-api`). Mitigação: CA3 resolve **com a Tray** no ticket antes de qualquer implementação; documentamos os dois entendimentos e o método server-side que usamos.
- **Conflito do `refresh_token` de uso único** (heziom-api × HeziomOS refrescam o mesmo app+loja). É risco de produção (token morto), não só de homologação. Mitigação: documentar aqui + apontar 38.2/38.12 (dono único). Não resolver nesta story.
- **`TRAY_API_HOST` da `heziom-api` preso à env (loja de teste hoje):** a ponte não é store-agnostic como o monorepo. Mitigação: registrar no pacote que a virada é troca de env no cutover (38.12); não hardcode nesta story.
- **Vazamento de segredo/PII nos logs do Netlify** ao capturar o console (a `heziom-api` loga `[webhook] ...` e chama a Graph API, que pode ecoar dados no erro). Mitigação: CA6 — mascarar + grep adversarial antes de anexar; captura na loja de teste (dados fictícios).
- **Observação de segurança (fora de escopo, sinalizar):** o webhook **primário** `webhooks-tray.js` **não valida** `X-Tray-Webhook-Secret` (só o fallback `tray-capi.js:48-55` valida). É um endpoint público sem segredo — levantar com @security como item separado (não é objeto desta story de documentação).
- **Multi-CD ativado sem sabermos:** se a conta ligar Multi-CD, o caminho single-stock do E38 fica incompleto. Mitigação: CA4 confirma com a conta e aponta 38.10.
