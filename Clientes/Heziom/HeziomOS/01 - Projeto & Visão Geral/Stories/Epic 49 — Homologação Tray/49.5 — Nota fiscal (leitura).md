## 🎯 Contexto

**Tarefa 09 do board de homologação da Tray** (projeto Odoo #1723, ticket #1842904): "API de nota fiscal". A Tray exige a evidência de que a aplicação **consome** a API de NF — print do código + print do console com a **URL completa** consumida.

**Estado hoje (verificado no código):** a aplicação **NÃO faz nenhuma chamada dedicada de nota fiscal.** O único dado de NF que existe vem **embutido na resposta do pedido**: `hub-tray-poll-orders/index.ts:71` (`const invoice = Array.isArray(order.Invoice) ? order.Invoice[0] : order.Invoice`) e `hub-tray-webhook/index.ts:125`, gravados em `hub.orders.invoice_number` / `invoice_key` (`0002_hub_schema.sql:192-193`, campos populados em `hub-tray-poll-orders/index.ts:84-85` e `hub-tray-webhook/index.ts:136-137`). Isso é o `number`+`key` que o pedido carrega — **não é o consumo do endpoint `/invoices`** que a Tray quer ver, e não traz os campos fiscais completos (série, chave de acesso de 44 dígitos, DANFE, status, XML).

**O que esta story entrega:** a **leitura dedicada** da NF-e — `GET {api_host}/orders/{order_id}/invoices` e `GET {api_host}/invoices/{invoice_id}` — pelo cliente store-agnostic (`_shared/tray.ts` `trayGet`, que respeita o token singleton `getValidToken` e herdará o limitador de 180/min da **49.4**), um parser puro e testado da NF, persistência no espelho fiel (`tray_mirror.invoices`) e a evidência de console (URL + payload, token mascarado) para a tarefa 09.

**Fronteira com o Epic 38 (não duplicar):** a **ESCRITA / emissão** de NF-e — `POST {api_host}/orders/{order_id}/invoices` — é **risco #1 do E38** (endpoint corrigido na spec, `epic-38-integracao-tray-completa.md:36`) e mora na **Story 38.7** (NF-e: vincular nota + conciliação fiscal Tray↔Literarius). Esta story cobre **só a LEITURA + a evidência**. A evidência de escrita para homologação (≥2 `POST` + ≥2 `PUT`, `epic-49-homologacao-tray.md:23`) sai de estoque (`PUT /products/:id`) + NF-e (`POST`, E38) — **não** desta story.

**Decisão de arquitetura (por que `_shared/tray.ts` e não `_shared/tray-auth.ts`):** existem dois clientes Tray no repo — o singleton store-agnostic (`_shared/tray.ts` `getValidToken`, host de `crm.tray_tokens.api_host`, `:60-69`) e o cache in-memory por env `TRAY_STORE_URL` (`_shared/tray-auth.ts:6-44`, usado pelas edges `hub-tray-*`). O princípio transversal do E49 (`epic-49-homologacao-tray.md:51`) manda **toda chamada nova** passar pelo `getValidToken` + limitador 49.4. Logo a leitura de NF usa o cliente de `_shared/tray.ts`.

## 🎯 Critérios de Aceite

- [ ] **CA1** — Helper de fetch dedicado em `_shared/tray-invoices.ts`: `getOrderInvoices(sb, tok, orderId)` chama `trayGet(sb, tok, \`/orders/${orderId}/invoices\`, {})` e `getInvoiceById(sb, tok, invoiceId)` chama `trayGet(sb, tok, \`/invoices/${invoiceId}\`, {})`. Usa exclusivamente o cliente de `_shared/tray.ts` (herda 401→refresh+retry `:129-131`, timeout 15s `:9`, e o limitador da 49.4). **Nada hardcoded** — `api_host` vem do token singleton.
- [ ] **CA2** — Parser puro `parseTrayInvoice(raw): TrayInvoice` (defensivo, no estilo `tray-products.ts:76-100`): extrai `number`, `series`, `access_key` (chave de 44 dígitos: `key`/`access_key`), `type` (0=entrada / 1=saída), `status`, `issue_date` (normalizada por `trayDate` de `tray.ts:25-30`), `value`, `danfe_url`, `has_xml` e **guarda o `raw`**. `unwrapInvoices(payload)` normaliza as formas `{Invoices:[{Invoice}]}` / `{Invoice}` / array / objeto único → `TrayInvoice[]`.
- [ ] **CA3** — Edge `crm-tray-invoice-read`: recebe `{ order_id }` (ou `{ invoice_id }`), auth = admin logado **OU** `service_role` **OU** cron (mesmo padrão de `crm-tray-sync-products/index.ts:30-31`), busca via CA1, parseia via CA2, faz **upsert** no espelho `tray_mirror.invoices` (raw + projeção, `onConflict: tray_invoice_id`) e retorna a(s) NF parseada(s). Erro → RFC 7807 (`internalError`/`badRequest` de `_shared/errors.ts`) + `reqId`.
- [ ] **CA4 (persistência + gap-filler)** — Tabela `tray_mirror.invoices` (migration idempotente, timestamp `> 20260730300000`): colunas projetadas + `raw jsonb` + `synced_at`/`updated_at`, seguindo o padrão de `20260701150000_crm_tray_products_catalog.sql:14`. Quando o `order_id` casa com uma linha de `hub.orders`, a edge **preenche `invoice_number`/`invoice_key` só se estiverem NULL** (gap-filler — nunca sobrescreve o que o poll/webhook já gravou; mesmo princípio da 20260730240000 do atendimento).
- [ ] **CA5 (evidência homologação)** — A edge loga no console, de forma capturável, a **URL completa consumida** (`GET {api_host}/orders/{id}/invoices?access_token=***`) e o **JSON de resposta** — com o `access_token` **mascarado** (`maskToken`) e a NF-e sem vazar PII em texto além do necessário. Content-Type = n/a (GET). Esse log é o print da tarefa 09 (leitura) no board.
- [ ] **CA6** — Testes `deno test` do parser (CA2) verdes, cobrindo: forma `{Invoices:[{Invoice}]}`, forma `{Invoice}` única, array direto, payload vazio → `[]`, `type` 0/1, chave de 44 dígitos vs. curta, data lixo (`0000-00-00` → null via `trayDate`), e ausência de XML/DANFE. `deno check` verde em todos os arquivos tocados.
- [ ] **CA7 (coordenação E38 — não duplicar)** — Nenhum `POST /orders/:id/invoices` (emissão NF-e) nesta story. A story referencia explicitamente a **38.7** como dona da escrita. Documentado no épico e no pacote de evidências (`docs/homologacao-tray-pacote.md`, seção "09 — API de nota fiscal") que a leitura é 49.5 e a escrita é 38.7.
- [ ] **CA8 (Security Gate)** — `@security *security-gate`: sem `access_token`/`consumer_secret` em log; RLS ENABLE+FORCE em `tray_mirror.invoices` + `service_role` all + SELECT só `authenticated` + REVOKE `anon` (padrão `20260710110000_tray_mirror_grants.sql:13-22`); mapa de PII fiscal (nome/CPF/CNPJ/endereço no `raw`) classificado; edge só service_role/admin/cron.

## 🔌 Contrato / endpoints

Base `{api_host}` = `https://lojatesteintegracaotray.commercesuite.com.br/web_api` na loja de teste 1501119 (lido de `crm.tray_tokens.api_host`, **nunca hardcoded**).

### Leitura (IN — esta story)
| Uso | Método | URL completa | Content-Type |
|-----|--------|--------------|--------------|
| NF-e de um pedido | GET | `{api_host}/orders/{order_id}/invoices?access_token={TOKEN}` | n/a (GET) |
| NF-e por id | GET | `{api_host}/invoices/{invoice_id}?access_token={TOKEN}` | n/a (GET) |
| Listagem (opcional/diagnóstico) | GET | `{api_host}/invoices?access_token={TOKEN}&limit=50&page=1` | n/a (GET) |

Construção da URL: `trayGet(sb, tok, path, params)` monta `${api_host}${path}?access_token=...&params` (`_shared/tray.ts:125-135`), com 401→refresh+retry (`:129-131`) e timeout 15s (`:9`).

**Resposta esperada (defensivo — validar contra a loja de teste antes do go-live):**
```jsonc
// GET /orders/{order_id}/invoices
{
  "Invoices": [
    { "Invoice": {
        "id": "987", "number": "000123", "serie": "1",
        "access_key": "3526...<44 dígitos>...9012",   // chave de acesso da NF-e
        "type": "1",                                    // 0=entrada, 1=saída
        "status": "issued",
        "date": "2026-07-14 10:22:31",                 // normalizar via trayDate()
        "value": "149.90",
        "xml": "<nfeProc>...</nfeProc>",
        "link_danfe": "https://.../danfe.pdf"
    } }
  ]
}
// GET /invoices/{invoice_id} → { "Invoice": { ...mesma forma... } }
```
> Os nomes de campo variam por versão da API da Tray — o parser é defensivo (fallbacks `key`/`access_key`, `serie`/`series`, `date`/`issue_date`, `link_danfe`/`danfe`) e **sempre guarda o `raw`** (re-mapear depois é barato), exatamente como `tray-products.ts:5-8`.

|-----|--------|-----|--------------|
| Emitir NF-e | POST | `{api_host}/orders/{order_id}/invoices` | `application/x-www-form-urlencoded` |

## ⚠️ Riscos

- **PII fiscal no `raw`** — a NF-e traz nome/CPF/CNPJ/endereço do destinatário. Mitigação: `tray_mirror.invoices` com RLS FORCE + SELECT só `authenticated` + REVOKE `anon`; log de evidência não deve despejar PII além do necessário (mascarar token; a captura de tela para homologação usa dado de teste). Security Gate obrigatório (CA8).
- **Loja de teste sem NF emitida** — o sandbox pode não ter NF-e emitida em nenhum pedido. Mitigação: emitir uma NF de teste (38.7/painel) ou, se bloqueado, registrar como pendente de reteste em produção (padrão CA-E7 do E38, `epic-38...:71`).
- **Forma do payload divergir da doc** — nomes de campo variam por versão. Mitigação: parser defensivo + `raw` fiel; validar contra resposta real (passo 4) antes do go-live.
- **Refresh token de uso único** — a chamada passa por `getValidToken`/`refreshTrayToken` (`tray.ts:36-58`); em paralelo com `heziom-api` há o conflito do E38 risco #2 / 38.2 (dono único do refresh). Não reintroduzir um 2º refrescador — reusar o singleton.
- **Cota 180/min** — leitura sob demanda (1 NF/pedido) fica muito abaixo, mas backfill em massa deve respeitar o limitador da **49.4**. Backfill fica OUT desta story.
- **Duplicar a escrita do E38** — risco de alguém adicionar o `POST` aqui. Mitigação: CA7 + referência explícita à 38.7; não abrir superfície de emissão nesta story.
