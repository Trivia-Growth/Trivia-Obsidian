## 🎯 Contexto

### O que a Tray exige (formulário de homologação — projeto Odoo #1723, ticket #1842904, Marcos M.)
Para **cada tarefa** do board (10 no total), a Tray pede **dois prints**: (1) o **código da aplicação** que consome a API e (2) o **console** mostrando a **URL consumida (completa)**, o **método**, o **Content-Type** (obrigatório em POST/PUT — `application/x-www-form-urlencoded` ou `application/json`) e o **JSON/payload** trafegado. É o formato descrito em `docs/homologacao-tray-pacote.md:8`.

### O problema hoje (por que uma story transversal)
1. **Não existe uma fonte única de "console".** As chamadas à Tray vivem espalhadas em várias edge functions (sync de catálogo, webhook, poll de pedidos) e **nenhuma loga a requisição de forma legível para print**. O wrapper `trayGet` (`_shared/tray.ts:125-135`) monta a URL com `access_token` embutido e **não loga nada** — em erro só devolve `${res.status}` truncado (`:133`). Ou seja: hoje capturar o "print do console" exigiria garimpar os logs da edge do Supabase, que não mostram a URL completa nem o payload de forma organizada.
2. **Risco de vazamento se logarmos ingênuo.** A URL da Tray carrega `?access_token=...` em **todo** GET (`tray.ts:127`) e `?consumer_key=...&consumer_secret=...` no `/auth` (`tray.ts:43`, `tray-auth.ts:21`). Logar a URL crua = **vazar credencial** no print que vai para um board de terceiro. Precisamos de **máscara** antes de qualquer `console.log`. (Regra da casa, reforçada pela memória do vazamento de token via erro da Graph API — `redactMetaError`.)
3. **Nem toda tarefa é chamada hoje.** Categoria (04), variação (06), imagem (07) e NF (09) hoje vêm **embutidas** na resposta de `/products` (`tray-products.ts:91`, `:96`, `:20-26`) ou **não são consumidas** (NF). As stories 49.1/49.2/49.3/49.5 criam as chamadas dedicadas; **esta story (49.6) é quem as exercita e captura a evidência** — e depende delas para as tarefas embutidas.

### Solução proposta
Um **edge function dedicado de homologação** — `crm-tray-homologacao` — que, **contra a loja conectada** (hoje a de teste **1501119**, host vindo de `crm.tray_tokens.api_host`; store-agnostic), dispara **em sequência** cada chamada do checklist e escreve no **console** um bloco padronizado e **mascarado** por tarefa. Os logs dessa execução (via `supabase functions logs crm-tray-homologacao`) **são** os prints de console que vão para o board. Um helper `maskTraySecrets` (testado) garante que `access_token`/`consumer_secret`/`refresh_token` e PII de pedido **nunca** aparecem. A função é **gated** (admin logado OU `service_role`), **read-only por padrão** (`dry_run` liga POST/PUT só quando explicitamente pedido) e **store-agnostic**. Fecha com o passo-a-passo de anexação por tarefa e mantém `docs/homologacao-tray-pacote.md` como o mapa mestre.

### Alternativas rejeitadas
- **Garimpar os logs das edges de produção** (sync/webhook/poll). Não mostram a URL completa nem o Content-Type de forma legível, e misturam tráfego real (PII de clientes reais). Um edge dedicado contra a **loja de teste** isola a evidência e controla o formato.
- **Logar dentro do `trayGet` genérico.** Poluiria toda a integração de produção com logs e arriscaria vazar token em prod. A captura fica **confinada** ao edge de homologação (só ele loga; usa o `trayGet` como está).
- **Print manual via Postman/cURL.** Não é reproduzível, não é store-agnostic e não prova que **a aplicação** consome a API (a Tray pede o print do **código da aplicação**). O edge usa exatamente o cliente `_shared/tray.ts` da aplicação.
- **Desmascarar "só para o print".** Proibido — o print vai para um board de terceiro. Máscara sempre; a Tray valida a integração pela **URL/estrutura**, não pelo valor do token (a doc deles inclusive orienta mascarar).

## 🔌 Contrato / endpoints

### Edge de homologação (nosso — dispara e loga)
`POST {SUPABASE_URL}/functions/v1/crm-tray-homologacao`
**Auth:** `Authorization: Bearer <JWT de admin>` **ou** `Bearer <SERVICE_ROLE_KEY>`. **Content-Type:** `application/json`.
**Body (todos opcionais):**
```json
{ "only": ["05","10"], "dry_run": true, "product_id": null, "order_id": null }
```
- `only` — subconjunto de tarefas (default: todas as disponíveis). `dry_run` — default `true` (sem POST/PUT). `product_id`/`order_id` — fixar o recurso do probe (senão pega o 1º do probe 05/10).
**Resposta 200:** `{ ok, ran: ["01","05","10",…], skipped: [{ "06": "helper 49.2 ausente" }], hint: "veja os logs: supabase functions logs crm-tray-homologacao" }`. O **valor** da evidência está no **console**, não no corpo HTTP.

### Chamadas à Tray disparadas pelos probes (doc oficial developers.tray.com.br)
Base `{api_host}` = `https://lojatesteintegracaotray.commercesuite.com.br/web_api` (loja teste; vem do `/auth`).

| Probe | Método | URL (mascarada no log) | Content-Type |
|---|---|---|---|
| 01 token (leitura/renova) | GET | `{api_host}/auth?refresh_token=***&consumer_key=***&consumer_secret=***` | n/a (GET) |
| 04 categoria | GET | `{api_host}/categories?access_token=***` | n/a (GET) |
| 05 produtos | GET | `{api_host}/products?access_token=***&limit=1&page=1` | n/a (GET) |
| 06 variação | GET | `{api_host}/products/{id}/variants?access_token=***` | n/a (GET) |
| 07 imagem (leitura) | GET | via `/products/{id}` (campo `ProductImage`) | n/a (GET) |
| 07 imagem (escrita, só `dry_run:false`) | POST | `{api_host}/products/{id}/images?access_token=***` | `application/x-www-form-urlencoded` |
| 09 NF | GET | `{api_host}/orders/{id}/invoices?access_token=***` | n/a (GET) |
| 10 pedidos | GET | `{api_host}/orders?access_token=***&limit=1` | n/a (GET) |

> Content-Type/payload só se aplicam a **POST/PUT** (regra do próprio formulário da Tray). Categoria/variação/NF/produtos são **GET** → o campo "Content-Type" fica `n/a (GET)` no print, e o payload é a **resposta** JSON mascarada. As evidências de **escrita** (POST/PUT — imagem 07, estoque `PUT /products/:id`, NF-e `POST /orders/:id/invoices`) saem do E38 e do `dry_run:false`; ver risco E38 #4 (mín. 2 POST + 2 PUT).

### Formato do bloco de console (exemplo — probe 05)
```
[HOMOLOG 05] GET https://lojatesteintegracaotray.commercesuite.com.br/web_api/products?access_token=***&limit=1&page=1
[HOMOLOG 05] content-type: n/a (GET)
[HOMOLOG 05] response(200): {"Products":[{"Product":{"id":"123","name":"Livro X","price":"49.90","ProductImage":[{"https":"https://.../img.jpg"}],"Variant":[...],"category_name":"Livros"}}],"paging":{...}}
```

## ⚠️ Riscos & edge cases

1. **Vazamento de segredo no print (CRÍTICO).** É o risco central. Mitigação: **todo** `console.log` passa por `maskTraySecrets`/`maskTrayJson`/`maskOrderPii`; teste property-based garante não-vazamento; Security Gate faz `grep` adversarial numa execução real. `Deno.env` nunca logado.
2. **PII de cliente real** em `/orders*`/`/invoices`. Mitigação: `maskOrderPii` mascara CPF/e-mail/telefone/nome; a captura roda na **loja teste** (dados fictícios), reduzindo exposição.
3. **`refresh_token` de uso único** (`tray.ts:32`). Probe 02 fora do default; quando rodar, usa `refreshTrayToken` que **persiste** o novo par — não deixa a sessão órfã. Nunca chamar o `/auth?refresh_token` cru.
4. **Dependência das stories 49.1–49.5.** Probes de categoria/variação/imagem/NF/limitador só produzem evidência quando o helper existir → `SKIP` gracioso; a story fecha as tarefas disponíveis primeiro e as demais conforme cada helper merge.
5. **Limite 180 req/min** (`epic-49:19`). O edge dispara **poucas** chamadas (1 por probe); rodar `only` em lotes evita rajada. Não há limitador global hoje (49.4).
6. **`dry_run` acidental em `false`** dispara POST/PUT reais (imagem/estoque/NF) na loja. Default é `true`; escrita exige o flag explícito e roda **só** na loja teste.
7. **Função exposta.** Se ficar pública, vira oráculo de token. Mitigação: gate admin/service_role + `verify_jwt=true` (não é webhook — ver `reference_heziomos_edge_verify_jwt_config`).
8. **Board Odoo sem integração.** Anexação é manual; nenhuma automação posta nada externo sem aprovação humana.
