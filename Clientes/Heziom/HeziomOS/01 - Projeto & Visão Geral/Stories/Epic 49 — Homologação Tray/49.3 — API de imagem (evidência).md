## 🎯 Contexto

A Tarefa 07 do board de homologação (Odoo #1723, ticket #1842904 — Marcos M., Suporte Técnico) pede evidência de que a integração consome a **API de imagem** da Tray: print do **código** + print do **console** com a **URL completa**, o **Content-Type** (só em POST/PUT) e o **payload/JSON** (token mascarado).

**Como está hoje (verificado no código):** a integração **NÃO chama** nenhum endpoint dedicado de imagem. As imagens chegam **embutidas na resposta de `/products`** e são lidas defensivamente:
- `supabase/functions/_shared/tray-products.ts:19-26` — `extractImages(p)` lê `p.ProductImage ?? p.images ?? p.Images` e extrai `https ?? http ?? url ?? image` de cada item.
- `supabase/functions/_shared/tray-products.ts:94` — `images: extractImages(p)` no espelho `tray_mirror.products`.
- `supabase/functions/_shared/tray-products.ts:115` — `image_url: m.images.length ? m.images[0] : null` na projeção `crm.products` (é a foto que o agente de vendas usa).
- A origem do payload é `crm-tray-sync-products/index.ts:47` → `trayGet(sb, tok, "/products", { limit: 50, page })` (wrapper GET em `_shared/tray.ts:125-135`).

**O nó da tarefa — leitura vs. escrita.** No catálogo oficial da Tray (developers.tray.com.br, 15/07) o endpoint **dedicado** de imagem é de **ESCRITA**: `POST /products/:id/images` (registrar), `POST /products/:id/variants/:variant_id/images`, `POST /products/images/remove` (remover). A **leitura** de imagem **é feita pelo produto** (campo `ProductImage`) — é exatamente o que a Heziom já faz. A Heziom **espelha** o catálogo (LÊ) e **não gerencia imagens** (não faz upload/troca de foto pela API). Portanto:
- Para **leitura**, a evidência natural é o `GET /products` mostrando o array `ProductImage` (o Content-Type/payload "só se aplica em POST/PUT", logo não há corpo a mostrar num GET).
- Um `POST` só entra **se a Tray exigir** evidência de escrita da API de imagem. **Não inventar** um fluxo de upload de imagem no sync — a Heziom não escreve imagens em operação. Se exigido, o POST é uma **demonstração isolada de homologação** (loja de teste, com remoção da evidência ao final), nunca parte do fluxo operacional.

Esta story define **a evidência mínima que satisfaz a Tarefa 07** e deixa explícito o que é leitura (fazemos) e o que é escrita (só se exigido, e como demonstração).

## 🎯 Critérios de Aceite

- [ ] **CA1 — Evidência de LEITURA por produto (caminho principal).** Rodar o `GET /products` real contra a loja de teste **1501119** e capturar no console a **URL completa** (`{api_host}/products?access_token=***&limit=50&page=1`) e o JSON de ≥1 produto que tenha o array **`ProductImage`** com pelo menos uma URL `https`/`http`. Anexar ao pacote (49.6) junto com o print do código `extractImages` (`tray-products.ts:19-26`) e a persistência (`:94` espelho, `:115` `crm.products.image_url`). Token/consumer **mascarados**. `Content-Type`: n/a (GET, sem corpo).
- [ ] **CA2 — Sondar leitura dedicada `GET /products/:id/images`.** Testar na loja 1501119 se `GET {api_host}/products/{id}/images?access_token=***` responde `200` com a lista de imagens do produto:
  - **Se 200** → adicionar helper dedicado `trayGetProductImages(sb, tok, productId)` (mesmo padrão de `trayGet` e das stories 49.1/49.2) e usar **essa** chamada como evidência "API de imagem" (URL completa + JSON no console). É a evidência mais limpa (endpoint próprio de imagem).
  - **Se 404/405/erro** → registrar no pacote que, na versão de API da loja, **a leitura de imagem é via `/products`** (não há GET dedicado) e a **evidência do CA1 é suficiente** para a Tarefa 07. Não forçar um endpoint inexistente.
- [ ] **CA3 — Decidir com a Tray se a Tarefa 07 exige ESCRITA (POST).** Perguntar no ticket #1842904 se a evidência de "API de imagem" precisa demonstrar `POST /products/:id/images` (registro) ou se a leitura basta. **Documentar a resposta** no `docs/homologacao-tray-pacote.md` §07. Enquanto não exigido, **não implementar** escrita de imagem (a Heziom não gerencia imagens pela API — só espelha).
- [ ] **CA4 — (Condicional a CA3 = "exige POST") Demonstração de homologação isolada.** Uma edge/script **guardado** (flag/env desligada por padrão, `service_role`, **só** loja de teste 1501119) que: (a) faz `POST {api_host}/products/{id}/images` registrando uma imagem de teste num produto de teste; (b) captura o console com **URL completa**, **`Content-Type: application/x-www-form-urlencoded`** e o **payload** (token mascarado); (c) **remove** a imagem criada em seguida (`POST {api_host}/products/images/remove`) para **não deixar resíduo** no catálogo. Nunca roda em produção (assert de `api_host` = host de teste). Requer um helper `trayPostForm` (não existe hoje — ver Riscos; coordenar com 49.5, que também precisa de POST).
- [ ] **CA5 — Higiene de token e limite.** Toda chamada nova respeita o token singleton (`getValidToken`, `_shared/tray.ts:60-69`) e o **limitador de 180 req/min** da 49.4 (a leitura dedicada de imagem por produto pode multiplicar chamadas — não estourar a cota). Nenhum `access_token`/`consumer_secret` aparece nos prints ou logs — **mascarar** (`***`). **Store-agnostic:** `api_host` sempre de `crm.tray_tokens`, nada hardcoded.
- [ ] **CA6 — Security gate.** `@security *security-gate` PASS/CONCERNS. Foco: (a) não vazar token em querystring de log; (b) no ramo POST — a escrita numa loja de 3º está atrás de flag + `service_role` + assert de loja de teste + remoção da evidência; sem upload de conteúdo arbitrário em prod.

## 📦 Escopo

**IN:**
- Evidência de **leitura** de imagem (caminho principal): captura real do `GET /products` mostrando `ProductImage` + print do código (`extractImages` e persistência).
- Sondagem + (se existir) helper `trayGetProductImages` para `GET /products/:id/images` como evidência dedicada.
- Decisão documentada (leitura basta × exige POST), com a Tray, no pacote.
- **Condicional:** demonstração isolada de `POST /products/:id/images` + `POST /products/images/remove` (guardada, loja de teste, com cleanup), **só se** a Tray exigir escrita.

**OUT:**
- Gerenciar imagens de produto pela API em operação (upload/troca de foto no sync) — a Heziom não faz isso; não criar esse fluxo.
- Imagens de variação (`POST /products/:id/variants/:variant_id/images`) — fora do escopo desta tarefa (variação é a 49.2); só citar se a Tray pedir.
- Reprocessar/otimizar/baixar as imagens do espelho — não é homologação.
- O limitador em si (é a 49.4) — aqui só **consumir** o limitador.

## 🔌 Contrato / endpoints

`{api_host}` vem de `crm.tray_tokens.api_host` (teste = `https://lojatesteintegracaotray.commercesuite.com.br/web_api`). **Nada hardcoded.**

### Leitura (o que a Heziom faz — evidência principal)
```
GET {api_host}/products?access_token={TOKEN}&limit=50&page={N}
Content-Type: n/a (GET, sem corpo)
→ { "Products": [ { "Product": {
      "id": "123", "name": "...",
      "ProductImage": [ { "id": "...", "https": "https://.../foto.jpg", "http": "http://.../foto.jpg", "thumbs": [...] } ],
      ... } } ] }
```
Código: `crm-tray-sync-products/index.ts:47` (chamada) · `_shared/tray-products.ts:19-26` (parse `extractImages`) · `:94` (espelho) · `:115` (`crm.products.image_url`).

### Leitura dedicada (SONDAR — só se a versão da loja suportar — CA2)
```
GET {api_host}/products/{id}/images?access_token={TOKEN}
Content-Type: n/a (GET, sem corpo)
→ (a confirmar contra a loja 1501119: lista de imagens do produto)
```
Helper novo (se 200): `trayGetProductImages(sb, tok, productId)` reusando o padrão `trayGet` (`_shared/tray.ts:125-135`: build da URL + 401→refresh+1 retry + timeout 15s).

### Escrita (CONDICIONAL — só se a Tray exigir — CA4; NÃO é fluxo operacional)
```
POST {api_host}/products/{id}/images
Content-Type: application/x-www-form-urlencoded
payload (form): Image[][http]=https://.../foto-teste.jpg    (URL da imagem; ou base64 — confirmar campo exato na doc)
→ { "id": "<image_id>", ... }
```
Remoção da evidência (limpar o resíduo logo após o print):
```
POST {api_host}/products/images/remove
Content-Type: application/x-www-form-urlencoded
payload (form): id=<image_id>   (id(s) da imagem criada no passo anterior — confirmar chave exata na doc)
```
> Payload/campo do POST marcados **"confirmar na doc oficial"** — a Draft não fixa formato que ainda não validamos. O POST exige um helper `trayPostForm` que **não existe** hoje em `_shared/tray.ts` (só há `trayGet`, GET-only). Coordenar com a 49.5 (NF-e), que também precisa de POST.

## ⚠️ Riscos

- **Confusão leitura×escrita (risco central):** o endpoint dedicado da Tray é POST (escrita). Se aceitarmos "implementar a API de imagem" ao pé da letra, criaríamos um fluxo de upload que **não existe** e não queremos. Mitigação: CA3 fixa a decisão **com a Tray** antes de qualquer POST; o POST (se houver) é demonstração, não operação.
- **`GET /products/:id/images` pode não existir** na versão de API da loja (o catálogo oficial lista só POST para imagem). Mitigação: CA2 trata os dois casos; o CA1 (leitura via produto) é a evidência garantida.
- **POST numa loja de 3º deixa resíduo** (imagem de teste no catálogo). Mitigação: `remove` imediato + flag + loja de teste + assert de host.
- **Não há helper de POST** em `_shared/tray.ts` (só `trayGet`). Mitigação: `trayPostForm` compartilhado com a 49.5 — não duplicar.
- **Vazamento de token no print/log** (querystring com `access_token`). Mitigação: mascarar `***` sempre; regra transversal do E49.
- **Estouro do limite de 180/min** se a leitura dedicada de imagem virar 1 chamada por produto no catálogo inteiro. Mitigação: só usar `trayGetProductImages` para **capturar evidência** (poucos produtos), não no sync de catálogo; respeitar 49.4.
