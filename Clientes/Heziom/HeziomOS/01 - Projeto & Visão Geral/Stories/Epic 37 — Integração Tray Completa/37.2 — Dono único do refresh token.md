# 37.2 — Dono único do refresh token

## Contexto
> **Risco crítico de produção.** O refresh token da Tray é de **uso único**: ao renovar, o par anterior é invalidado. Hoje DOIS sistemas renovam o token da MESMA loja — a `heziom-api` (cron 2h) e o HeziomOS (sync/refresh proativo). Em produção, um invalida o token do outro → integração cai de forma intermitente e difícil de diagnosticar. Já registrado como risco em aberto na [[STORY-017 — Integração Tray]].

## Spec de Referência
- [[Tray — Auditoria de Capacidades vs Produção]] (Risco crítico do token)
- [[Tray - Autenticação]] (access 3h / refresh 30 dias; refresh uso único)
- [[heziom-api — Referência Técnica]] (tray-token-refresh.js, cron 2h)

## Decisão a tomar (registrar em ADR)
Escolher UM modelo:
- **(A) HeziomOS é dono** (recomendado): `tray_tokens` no Supabase é a fonte única; a `heziom-api` passa a só **ler** o access_token (nunca refresca). O refresh vira responsabilidade de um único componente do HeziomOS.
- (B) Lock distribuído: ambos coordenam via advisory lock/linha de controle antes de refrescar, com o novo par gravado atomicamente e relido pelo outro.

## Escopo técnico
- Implementar o modelo escolhido; desligar o refresh do lado não-dono.
- Gravação atômica do novo par (access+refresh) + `refreshed_at`; leitura sempre da fonte única.
- Margem de renovação proativa (ex.: renovar com <1h do access) num só lugar.
- Telemetria: log de cada refresh (quem, quando) para auditar que só o dono renova.

## Critérios de Aceite
- [ ] CA1 — Exatamente UM componente refresca o token por loja (o outro só lê).
- [ ] CA2 — Simulação de refresh concorrente NÃO invalida o token em uso.
- [ ] CA3 — Com heziom-api + HeziomOS ativos na loja de teste por ≥24h, o token nunca expira sem renovar (0 respostas 401 por token inválido).
- [ ] CA4 — ADR registrado com o modelo escolhido e o racional.
- [ ] CA5 — Escrita do novo par é atômica (sem janela em que um leia par velho já invalidado).

## Prontidão para produção
Modelo store-agnostic: mesmo dono/lock vale para a loja 1345958. Documentar no runbook (37.1) que, no cutover, o seed inicial do token de produção é feito pelo dono.

## Dependências
- Coordenação obrigatória com o repo `heziom-api`.
- Precede o agendamento de qualquer cron de sync em produção.
