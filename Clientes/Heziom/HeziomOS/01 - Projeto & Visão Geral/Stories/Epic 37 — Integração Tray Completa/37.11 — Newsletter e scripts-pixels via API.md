# 37.11 — Newsletter e scripts/pixels via API

## Contexto
> Duas capacidades de marketing que a API oficial confirma e que o vault não mapeava: **Newsletter** (`/newsletters`, double opt-in) para capturar leads da vitrine para o e-mail marketing, e **Scripts externos** (skill `tray-scripts-externos`) para injetar pixels/tags (Meta, GA4, remarketing) sem editar o tema.

## Spec de Referência
- [[Tray — Auditoria de Capacidades vs Produção]] (Newsletter, Scripts externos)
- [[Tray - Carrinho Abandonado e Scripts]] (casos de uso de scripts)

## Escopo técnico
- Newsletter: `GET /newsletters`, `POST /newsletters` (cadastra), `POST /newsletters/confirm` (double opt-in). Exportar assinantes → ferramenta de e-mail da Heziom (respeitando consentimento; regra lead×cliente).
- Scripts: gestão de scripts externos via API (pixels Meta/GA4, tags de campanha, remarketing) — com liga/desliga por flag; fallback = configuração manual no admin da Tray.

## Critérios de Aceite
- [ ] CA1 — Assinante de newsletter capturado na loja de teste aparece na base (com opt-in).
- [ ] CA2 — Export de assinantes alimenta o e-mail marketing respeitando consentimento.
- [ ] CA3 — Um pixel/tag é injetado via API na loja de teste (ou, se o endpoint não responder, documentado o fallback manual no admin).
- [ ] CA4 — Scripts têm liga/desliga; teste não quebra a vitrine.

## Prontidão para produção
Store-agnostic. **Menor prioridade** — pode ir depois do go-live. Confirmar na loja de teste se o recurso de scripts externos responde por API (a doc oficial confirma a skill, mas o caminho exato pode exigir validação); se não, seguir por admin manual.

## Dependências
- 37.3 (endpoints), integração com o motor de e-mail (E20) para o destino dos leads de newsletter.
