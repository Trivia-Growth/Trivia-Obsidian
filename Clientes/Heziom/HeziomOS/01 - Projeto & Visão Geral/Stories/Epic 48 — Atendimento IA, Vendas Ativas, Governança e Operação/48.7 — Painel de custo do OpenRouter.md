# Story 48.7: Painel de custo da IA de atendimento (OpenRouter) no dashboard

**Epic:** Epic 48 — Atendimento IA: Vendas Ativas, Governança e Operação
**Story ID:** 48.7
**Priority:** 🟡 Medium
**Status:** Draft
**Type:** ✨ Feature
**Security gate:** ✅ Sim — custo é "dado de operação" restrito a admin (a edge atual corrige um IDOR de métrica); nova superfície de agregação de custo precisa manter esse gate e não vazar consumo por conversa/agente a quem não pode ver.
**Estimativa:** M — o dado **já é gravado** (`crm.ai_token_usage`) e há edge/RPC de custo por-agente. O trabalho é uma **RPC/edge de agregação** (todos os agentes + série temporal + por modelo) e um **painel** no dashboard; a atribuição "IA de atendimento" já é limpa. Fase B opcional: capturar o **custo real do OpenRouter** (hoje é estimativa).

---

## 🎯 Contexto

### Como funciona hoje (ponta a ponta)
1. **O consumo já é gravado** — `crm.ai_token_usage` (`supabase/migrations/20260701140000_crm_ai_token_usage.sql:12-22`): `agent_id` (FK `crm.ai_agents`), `conversation_id`, `model`, `input_tokens`, `output_tokens`, `cached_input_tokens`, `cost_usd NUMERIC(12,6)`, `created_at`. Índices por `(agent_id, created_at)` e `(created_at)` (`:23-25`). **Não há coluna `provider`.** RLS FORCE; SELECT a `authenticated`, escrita só `service_role` (`:49-62`).
2. **Quem escreve:** só o orchestrator de atendimento — `logTokenUsage()` em `crm-ai-orchestrator/index.ts:575` (único `insert` de `ai_token_usage` no repo). Ou seja, **`ai_token_usage` = custo da IA de atendimento**, atribuição limpa. O Estúdio de Conteúdo (E40) usa tabela **própria** (`crm.content_ai_costs`, `20260729100000:132-141`) — não polui.
3. **O custo é ESTIMATIVA local, não o real do OpenRouter** — `cost_usd = estimateCostUsd(model, usage)` (`_shared/pricing.ts:40-44`): tokens **reais** (do `usage` da API) × **tabela de preços estática** de 3 modelos + fallback `DEFAULT_PRICE` (Sonnet), `pricing.ts:16-21` ("APROXIMADOS", `:1-4`). O código lê `usage.prompt_tokens/completion_tokens/cached_tokens` (`index.ts:479-486`; `_shared/ai-stream.ts:218-224,283`) mas **nunca** pede o custo real do OpenRouter (`usage:{include:true}` no body ou `GET /api/v1/generation?id=`). `USD_BRL=5.4` fixo (`pricing.ts:7`).
4. **RPC/edge de custo existentes:** RPC `crm.ai_month_cost(p_agent_id, p_since)` (soma `cost_usd`, service_role-only, `20260701140000:65-74`); edge `crm-ai-usage-summary` (**admin-only** via `is_admin`, `index.ts:36-37`; **por agente**, `agent_id` obrigatório `:27-28`; **mês corrente** `:39`; retorna `{month_cost_usd, responses, avg_cost_usd, budget_usd, alert_pct}` `:50-56`). Teto/alerta por agente: `crm.ai_agents.monthly_budget_usd`/`budget_alert_pct` (`20260701140000:29-36`; aplicados no orchestrator `:892-907`, alerta Teams `:348-373`).
5. **UI hoje:** `BudgetCard` na aba Operação (`AgentOperationTab.tsx:463,470-483,518-544`) — "Gasto no mês" + respostas + média + barra do teto, **por agente**. Não há visão agregada nem série temporal.

### O defeito / a lacuna
1. **Não há painel de custo consolidado** para acompanhar: só existe o card por-agente/mês-corrente. Falta **total de todos os agentes**, **série temporal** (dia/semana/mês), **breakdown por modelo**, **filtro de período**.
2. **O custo é estimado, não o real do OpenRouter** — para "acompanhar o custo do OpenRouter" de forma fiel, o ideal é capturar o custo real da API (hoje diverge por preços desatualizados, cache-write não capturado, modelos fora da tabela caindo no fallback Sonnet).
3. **Sem coluna `provider`** — não dá para separar "quanto foi OpenRouter vs Anthropic nativo" dentro da tabela (hoje todos os agentes de prod usam openrouter, mas isso não está gravado na linha de custo).

### Solução proposta
- **Fase A — Agregação + painel (MVP, usa o dado que já existe):**
  - **RPC `crm.rpc_ai_cost_overview(p_from, p_to, p_bucket text)`** (`SECURITY DEFINER`, `service_role`, fuso `America/Sao_Paulo`): total `cost_usd` + tokens no período, **série temporal** por `date_trunc(p_bucket, created_at)`, **por agente** e **por modelo**, nº de respostas. Reusa os índices de tempo (`20260701140000:25`).
  - **Edge `crm-ai-cost-panel`** (ou estender `crm-ai-usage-summary`) — **admin-only** (mantém o gate atual, que corrige IDOR de métrica), aceita período arbitrário e bucket. Devolve o overview + comparação com os tetos (`monthly_budget_usd` somados).
  - **Painel no dashboard** — card/aba "Custo da IA de atendimento (OpenRouter)": total do mês + tendência (linha por dia/semana), quebra por agente (Helena etc.) e por modelo, % do teto agregado. Placement: aba **Operação** em `/atendimento/config` (admin, ao lado do `BudgetCard`) ou uma sub-aba de custo. Moeda: USD (com opção BRL usando um câmbio configurável — hoje `USD_BRL=5.4` fixo).
- **Fase B — Custo real do OpenRouter (fidelidade, opcional nesta story):**
  - Capturar o custo real por chamada: enviar `usage:{include:true}` no body (OpenRouter retorna `cost`) **ou** consultar `GET /api/v1/generation?id={gen_id}` (`total_cost`) e persistir em coluna nova `cost_usd_real` (+ opcional `provider`) em `ai_token_usage`. O painel passa a mostrar **real** (com o estimado como fallback quando faltar). Fecha o gap de fidelidade que o JG quer para "acompanhar o custo do OpenRouter".

### Alternativas rejeitadas
- **Só reusar o `crm-ai-usage-summary` por-agente na tela.** Não dá visão consolidada nem tendência — é o que já existe e não atende "acompanhar". Precisa de agregação.
- **Contar/somar custo no cliente.** Viola o teto de 1000 e o padrão da casa; e custo é admin-only. Agregação por RPC service_role.
- **Colocar o painel em `/analytics` (manager+).** A edge/RPC de custo são admin-only de propósito (IDOR de métrica, `crm-ai-usage-summary:34-37`); expor a manager exigiria afrouxar o gate. Manter **admin**.

## 🗃️ Modelo de dados

**Fase A: sem DDL** (só RPC de leitura sobre `crm.ai_token_usage`). **Fase B (opcional): DDL leve idempotente** (timestamp DEPOIS de `20260729190000`, ex.: `20260730170000_crm_ai_token_usage_real_cost.sql`):
```sql
ALTER TABLE crm.ai_token_usage
  ADD COLUMN IF NOT EXISTS cost_usd_real numeric(12,6),   -- custo real do OpenRouter (NULL = usar cost_usd estimado)
  ADD COLUMN IF NOT EXISTS provider      text;            -- 'openrouter' | 'anthropic' | ...
COMMENT ON COLUMN crm.ai_token_usage.cost_usd_real IS
  'Custo real informado pela API (OpenRouter usage.cost / generation.total_cost). NULL cai no cost_usd estimado.';
```
**RLS / grants:** RPC `SECURITY DEFINER SET search_path=crm,public`, `REVOKE ALL`/`GRANT service_role`; a edge aplica `is_admin`. Colunas da Fase B herdam a RLS de `ai_token_usage` (SELECT a authenticated, escrita service_role).

## ⚠️ Riscos & edge cases

1. **Custo estimado ≠ real:** deixar explícito no painel (rótulo "estimado") até a Fase B; senão o número engana.
2. **RBAC:** manter admin-only (custo é dado de operação; a edge atual corrige IDOR). Não expor a manager sem decisão.
3. **Preços/câmbio estáticos:** `pricing.ts` só 3 modelos + fallback Sonnet e `USD_BRL=5.4` fixo — documentar; a Fase B (custo real) resolve a fidelidade.
4. **Sem `provider` hoje:** segmentar OpenRouter vs nativo só na Fase B (coluna nova); no MVP, por `model`.
5. **Volume de `ai_token_usage`:** é linha-a-linha por interação; para tendência longa, considerar rollup/materialized (follow-up), mas os índices de tempo já suportam o MVP.
6. **Convenção de migration (Fase B):** timestamp DEPOIS de `20260729190000`; idempotente.
