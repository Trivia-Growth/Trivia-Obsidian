# Story 48.6: Centralizar a configuração da IA no front (cron + mensagem de follow-up e afins, hoje só no backend)

**Epic:** Epic 48 — Atendimento IA: Vendas Ativas, Governança e Operação
**Story ID:** 48.6
**Priority:** 🟡 Medium
**Status:** Draft
**Type:** ✨ Feature + 🔧 Refactor
**Security gate:** ✅ Sim — expõe no front controles que hoje são só de infra (status/toggle do **cron**, que carrega `CRON_SECRET`), torna editável a **diretiva de follow-up** (texto que a IA manda ao cliente), grava em `crm.ai_agents` (admin-only, superfície RLS). Não pode vazar `CRON_SECRET` ao client.
**Estimativa:** M — a superfície de config já existe (`AtendimentoConfig.tsx`, 15 abas); o trabalho é: coluna `follow_up_directive` + campo na UI; um edge service_role de **status do cron** (sem vazar segredo) + toggle; e consolidar/clarear as abas IA+Operação. Sem tabela nova.

## 🎯 Contexto

### Como funciona hoje (ponta a ponta)
Superfície principal: **`AtendimentoConfig.tsx`** (rota `/atendimento/config`), 15 abas via `TABS` (`:65-167`). Config transversal fica em `Settings.tsx:826-874`. Abas de IA relevantes:
- **IA** (`AISettingsTab.tsx`): identidade (nome, provider, model, temperature, max_tokens, behavior, **system_prompt**), número vinculado, company/sdr, horário, `hours_mode`, `out_of_hours_message`, `handoff_message`. Grava `crm.ai_agents` (`:451-488`).
- **Operação** (`AgentOperationTab.tsx`): flags do motor (tool_use, RAG, sales, tom regional), **follow-up (liga/desliga + aguardar-min + máx-por-conversa)** (`:294-337`), regras 17.7 (limite diário, transferir após N turnos, palavras de escalonamento), orçamento, Lições Aprendidas, especialistas. Grava `crm.ai_agents` (`:117-119`) + `agent_corrections` (`:592-599`).

**Config que está SÓ no backend / hardcoded (a queixa do JG):**
1. **Cron de follow-up — sem UI.** O job `crm-ai-followup-sweep-15min` (`*/15 * * * *`) é gerenciado por SQL/Management API (runbook `vindi-links-pagamento.md:160-190`; kill-switch `cron.alter_job(..., active := true/false)` `:166-172`; (re)criação `cron.schedule(...)` `:178-187`). A edge é `crm-ai-followup-sweep` (`config.toml:76-79`, `verify_jwt=false`, auth por `x-cron-secret`). **Nenhuma migration cria o cron.** A única alavanca no front é o toggle `follow_up_enabled` **por agente** (`AgentOperationTab.tsx:329-337`) — que **NÃO liga/desliga o scheduler**, só a elegibilidade da conversa. **Se o pg_cron estiver `active=false`, nenhum follow-up sai e isso é invisível na UI** (foi a classe de silêncio que escondeu os 3 bugs da E47).
2. **Mensagem/diretiva de follow-up — hardcoded, sem campo.** `FOLLOWUP_DIRECTIVE` é constante em `crm-ai-orchestrator/index.ts:127-132`, usada em `:1203` (`messages.push({ role:'user', content: FOLLOWUP_DIRECTIVE })`). É **diretiva de LLM, não texto literal** (o comentário `:126`/`:1200-1201` deixa explícito: "meta-instrução, não fala do cliente") — o modelo **gera** a mensagem a partir dela. Tools ficam OFF no follow-up (`:1217-1219`). **Não há coluna nem UI** para editar o texto/tom (grep por `follow_up_prompt/follow_up_message/followup_directive` = vazio).
3. **Batch do sweep — env, sem UI.** `FOLLOWUP_SWEEP_BATCH` (default 20, clamp 1..100, `crm-ai-followup-sweep/index.ts:23-26`).
4. **Hardcoded sem UI (fora do escopo mínimo, mas mapeado):** `SALES_GUARDRAILS` (`catalog-tool.ts:33-54`), `HEADER` do prompt (`_shared/prompt-builder.ts:63-70`), janela 24h (`wa-window.ts:12`), brand facts.

**Schema `crm.ai_agents`** (base `0001:795-816` + ~33 colunas): a maioria já editável na UI; **não existe** coluna para texto/tom do follow-up, guardrails, batch, agendamento. `follow_up_enabled_at` (cutover E47) é carimbo automático por trigger `trg_stamp_follow_up_enabled_at` (`20260729180000:41-45`), **não editável direto** (a UI não pode mexer nele à mão).

**RBAC:** `/atendimento/config` `minRole:'coordenador'` (`nav.ts:263-268`); abas sensíveis (IA/Operação/Conhecimento/canais) `minRole:'admin'`. INSERT/UPDATE de `crm.ai_agents` exige `crm.is_admin` (`20260710150000_crm_agent_config_admin_only.sql:9-21`); service_role bypassa.

### O defeito / a lacuna
1. **A mensagem/diretiva de follow-up não é editável no front** (hardcoded, global — nem por-agente).
2. **O cron não tem UI**: não dá para ver se está ligado, nem ligar/desligar/ajustar sem SQL — e o toggle por-agente engana (parece que liga, mas o scheduler pode estar off).
3. **Config espalhada**: IA e Operação gravam a mesma `crm.ai_agents` sem se falar; campos da mesma story (17.7) em abas diferentes; RAG (toggle em Operação) depende da base (aba Conhecimento) sem indicação cruzada.

### Solução proposta
- **Fase A — Diretiva de follow-up por-agente e editável:** coluna `crm.ai_agents.follow_up_directive text` (fallback para a constante quando `NULL`). O orchestrator passa a usar `agent.follow_up_directive ?? FOLLOWUP_DIRECTIVE` em `:1203`. Campo na aba Operação, junto do bloco de follow-up (`AgentOperationTab.tsx:294-327`), com texto de ajuda deixando claro que é **diretiva** (a IA gera a mensagem a partir dela), não um template literal — e um preview do que ela instrui. (Opcional: oferecer também um "modo template literal" para quem quiser texto fixo.)
- **Fase B — Painel de status/controle do cron (sem vazar segredo):** edge `crm-followup-cron-status` (service_role, auth admin) que **lê** `cron.job`/`cron.job_run_details` do sweep (ativo?, última execução, sucesso/erro, próximos) e permite **toggle** `active` via `cron.alter_job` **do lado do servidor** — o `CRON_SECRET` **nunca** vai ao client. Card na aba Operação: "Motor de follow-up: ligado/desligado, última rodada há X, N disparos" + botão pausar/religar + link para o runbook. Deixa visível a diferença entre "flag do agente" e "scheduler ligado".
- **Fase C — Consolidar/clarear:** unificar visualmente as abas IA+Operação (ou ao menos linkar as dependências: RAG↔Conhecimento, número↔canal, `out_of_hours_message`/`handoff_message` junto do resto do comportamento). Expor `budget_alert_pct` (hoje só exibido, `AgentOperationTab.tsx:541`). Opcional: batch do sweep como campo (com clamp), se o JG quiser.

### Alternativas rejeitadas
- **Expor o `CRON_SECRET`/agendar cron pelo client.** Vazaria segredo e daria ao browser poder de infra. Rejeitada: toda operação de cron passa por **edge service_role**; o client só vê status e aperta um botão.
- **Cron em migration para "ter no código".** A casa proíbe cron em migration (carrega `CRON_SECRET`). Mantém via Management API + agora um edge de status/toggle.
- **Transformar `FOLLOWUP_DIRECTIVE` em texto literal enviado.** Perde a naturalidade (a diretiva gera mensagem contextual). Mantém como diretiva por-agente; template literal só como opção extra.
- **Reescrever toda a superfície de config numa aba só.** Escopo grande e arriscado; começar consolidando o essencial (follow-up completo num lugar) e clareando dependências.

## 🗃️ Modelo de dados

DDL idempotente (timestamp DEPOIS do maior existente, ex.: `20260730150000_crm_ai_agents_followup_directive.sql`):
```sql
ALTER TABLE crm.ai_agents
  ADD COLUMN IF NOT EXISTS follow_up_directive text;
COMMENT ON COLUMN crm.ai_agents.follow_up_directive IS
  'Diretiva de reengajamento do follow-up (meta-instrucao p/ o LLM; NULL usa o default FOLLOWUP_DIRECTIVE hardcoded).';
```
Sem tabela nova. O cron **não** vai para migration (segue via Management API; o edge de status apenas lê/altera `cron.*`). `follow_up_enabled_at` continua carimbado por trigger (a UI não o edita).

**RLS / grants:** `follow_up_directive` herda a RLS admin-only de `crm.ai_agents` (`20260710150000`); a edge de cron é service_role + auth admin; nada de `CRON_SECRET` no client. Não alargar papéis.

## 🔌 Contratos

### Edge nova: `crm-followup-cron-status`
`GET …/crm-followup-cron-status` (Bearer jwt, gate admin) → lê `cron.job` (jobid, schedule, `active`) + últimas `cron.job_run_details` do sweep. Response:
```json
{ "scheduled": true, "active": true, "schedule": "*/15 * * * *",
  "last_run": { "start": "…", "status": "succeeded" }, "last_fired": 5 }
```
`POST …/crm-followup-cron-status` body `{ action: 'pause'|'resume' }` → `cron.alter_job(jobid, active := …)` server-side. `CRON_SECRET` nunca exposto. Erros RFC7807: `403` não-admin; `500` reqId.

### Orchestrator (diretiva por-agente)
`crm-ai-orchestrator/index.ts:1203` passa a usar `agent.follow_up_directive ?? FOLLOWUP_DIRECTIVE` (o SELECT do agente já traz a coluna). Sem novo contrato de rede.

### UI → banco
`AgentOperationTab` grava `follow_up_directive` no patch de `crm.ai_agents` (junto do bloco de follow-up), RLS admin-only.

## ⚠️ Riscos & edge cases

1. **`CRON_SECRET` no client:** jamais expor; toda operação de cron é server-side (edge). A resposta de status não deve conter o segredo.
2. **Diretiva editável = superfície de prompt:** limitar tamanho e sanitizar; a diretiva vai ao LLM (não ao cliente direto), mas ainda pode degradar o comportamento — manter o default como fallback e validar mudanças (pode usar o `crm-ai-eval-corrections` como referência de não-regressão).
3. **Flag×scheduler:** o card precisa deixar claro que ligar a flag não liga o cron; senão repete o mal-entendido da E47.
4. **`follow_up_enabled_at`:** a UI não pode editar o carimbo de cutover (é do trigger); só liga/desliga a flag.
5. **Convenção de migration:** timestamp DEPOIS de `20260729190000`; idempotente.
6. **Admin-only:** manter a RLS de `crm.ai_agents`; não afrouxar para coordenador.
