- `2026-05-25` — **Fix do rebuild on publish — admin-publish quebrado há 3 dias.** JG reportou que ao gerar nova imagem de capa via IA pro post `mao-de-obra-terceirizada-sao-paulo` e atualizar no admin, a publicação não refletia em produção. Diagnóstico via `gh run list`: TODOS os runs do workflow `Deploy to Netlify` desde 22/05 estavam `failure` com `Error: Unauthorized: could not retrieve project` (secret `NETLIFY_AUTH_TOKEN` do GitHub Actions expirado). Mas o `trigger-rebuild` gravava `via:'github', ok:true` em `configs_seo[last_rebuild]` porque o GitHub `repository_dispatch` só retorna 204 quando aceita o evento — não sabe se o workflow vai rodar. Resultado: 4 dispatches consecutivos do admin em ~90s sem que nenhum build acontecesse. Por que o site continuava atualizando em pushes? O Netlify tem 2 mecanismos de deploy independentes: auto-deploy Netlify-GitHub (ouve `push`, funciona) e GitHub Actions deploy.yml (ouve `push` + `repository_dispatch`, quebrado). Admin-publish usava só o segundo. **Fix imediato**: empty push (`599d4d5`) acionou auto-deploy e imagem nova ficou visível. **Fix permanente** (`caf45e0`): edge function `trigger-rebuild` invertida — Netlify Build Hook agora é caminho primário, GitHub dispatch é fallback. Validado em produção pelo JG editando título do mesmo post: `via:'netlify-hook', ok:true`, site refletiu em ~1min. **Dívida pendente**: regenerar `NETLIFY_AUTH_TOKEN` no GitHub é opcional (workflow continua falhando em pushes mas auto-deploy + build hook cobrem).
